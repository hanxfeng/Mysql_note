介绍：
	锁是计算机协调多个进程或线程并发访问某一个资源的机制。
	MySQL中有：
		全局锁：锁定数据库中的所有表
		表级锁：每次操作锁住整个表
		行级锁：每次操作锁住对应的行数据
全局锁：
	介绍：
		对整个数据库实例加锁，加锁后整个实例处于只读状态。
		经典使用场景是做全库逻辑备份，对所有表锁定，获得一致视图，保证数据完整性
	语法：
		`flush tables with read lock;` 对数据库加上全局锁
		`unlock tables` 解除全局锁
	问题：
		1.主库备份时，备份期间不能执行操作
		2.从库备份时，备份期间不能从主库同步二进制日志，导致主从延迟
表级锁：
	介绍：
		每次操作锁定整张表，锁定力度大，发生锁冲突的概率最高，并发度最低。
		主要有表锁，元数据锁，意向锁
	表锁：
		分为两类：
			表共享读锁：
				当锁定表时当前会话只能读不能写，其他会话只能读不能写
			表独占写锁：
				当锁定表时当前会话能读能写，其他会话不能读也不能写
		语法：
			加锁：`lock tables 表名1 read/write , 表名2 read/write;`
			释放锁：`unlock tables;` 会释放所有锁，会话断开也会释放所有锁
	元数据锁（MDL）：
		元数据锁加锁的过程是系统自动控制的，无需显式使用，在访问一张表时会自动加上，元数据锁的主要作用是维护表元数据的数据一致性，在表上有活动事务时，不可以对元数据进行写入操作，避免DML与DDL冲突，保证读写正确性。
	意向锁：
		介绍：
			当执行了一个语句，为表增加了一个行级锁后，另一个会话要对表增加一个表锁，此时需要先检查表中有没有行锁，如果一行行检查效率太低，因此在的第一个会话在加行级锁的同时会为表加一个意向锁，此时第二个会话只需要检查有没有意向锁，意向锁是否兼容即可，也就是说，意向锁和元数据锁一样是自动添加的。
		分类：
			意向共享锁：
				由select ，  lock in share mode添加
				与表读锁兼容，表写锁互斥
			意向排他锁：
				由insert ,delete ,update,select ... for update添加 
				与读锁写锁都互斥，但与其他意向锁兼容
		查询意向锁：`select object_schema,object_name,index_name,lock_type,lock_data from performance_schema.datalocks;`
行级锁：
	介绍：
		每次操作锁定对应的行数据，锁定力度最小，发生锁冲突的概率最高，主要用于InnoDB引擎。
		InnoDB引擎的数据主要是根据索引组织的，行锁是对索引上的索引项加锁实现的，而不是对记录加锁。
	分类：
		行锁：
			锁定单个的行记录，防止其他事务进行delete和update，RC和RR隔离级别都支持
		间隙锁：
			锁定索引记录的间隙（可以理解为阻止在两个索引之间插入数据），确保索引记录间隙不变，防止其他事务在这个间隙进行insert,产生欢读，RR隔离级别下支持
		临键锁：
			行锁和间隙锁组合，同时锁住数据和前面的间隙，在RR隔离级别下支持
	行锁：
		共享锁;
			允许一个事务去读一行，阻止其他数据获得相同数据集的排他锁
		排他锁：
			允许获取排他锁的事务更新数据，阻止其他事务获取相同数据集的共享锁和排他锁
			这里相同数据集指同一行数据
		注意：
			InnoDB引擎的行锁是对索引加的锁，因此，如果不通过索引检索数据，行锁就会升级成表锁

|             SQL              | 行锁类型<br> |  说明  |
| :--------------------------: | :------: | :--: |
|            insert            |   排他锁    | 自动加锁 |
|            update            |   排他锁    | 自动加锁 |
|            delete            |   排他锁    | 自动加锁 |
|            select            |  默认不加锁   |      |
| select ...lock in share mode |   共享锁    | 手动指定 |
|    select ... for update     |   排他锁    | 手动指定 |
	间隙锁：
		1.对唯一索引进行等值查询时，若对象不存在，则优化为间隙锁
		2.对普通索引进行等值查询时，向右遍历时最后一个值不满足查询条件，则临键锁（next-key lock）退化为间隙锁
		解释：
			假设表`user`有普通索引`age`，索引值为：10、20、30。
			现在执行等值查询：`select * from user where age = 25 for update;`（加锁查询）
			1. **InnoDB 的遍历过程**：
			要找`age=25`，但索引中没有 25，InnoDB 会向右遍历索引：
		    先看 20（小于 25，继续向右）；
			再看 30（大于 25，停止遍历）。
	        这里 “最后一个遍历到的值” 是 30，而 30≠25（不满足查询条件）。
			2.**锁的变化**：
		    原本，临键锁会锁定范围是 (20,30]（既锁间隙 (20,30)，又锁记录 30）。
		    但因为最后遍历到的 30 不满足`age=25`，InnoDB 会 “退化” 锁：只锁间隙 (20,30)，不锁记录 30。
		    此时，临键锁就退化为了间隙锁。
		3.对唯一索引进行范围查询时，会对所有满足条件的数据加行锁，然后在最后一个满足条件的数据上加临键锁，锁住该行数据和其右侧间隙
		4.间隙锁是防止其他事务在间隙中插入数据，不会组织其他事务为同一个间隙加上间隙锁
