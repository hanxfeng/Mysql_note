插入数据优化：
	insert优化：
		1.批量插入
			因为insert插入时每次都会与数据库连接进行传输，性能较低
			一次性插入多条数据尽量用一条插入，但不要超过一千条
		2.手动事务提交
			MySQL事务是默认开启的，因此一条条提交会反复开启关闭事务，手动开启事务，所有数据上传完毕后再提交事务效率更高
		3.主键顺序插入
			顺序插入主键效率要高于乱序插入
		4.大批量插入数据
			如果需要一次性插入大量数据，insert插入性能较低，可以使用load指令，将本地excel之类的文件数据录入数据库
			`load data local infile '文件路径' into table 表名 fields teminated by '字段分隔符' lines teminated by '行分隔符';`
主键优化：
	数据组织方式：
		在InnoDB引擎中，表数据是根据主键顺序组织存放的，这种存放方式成为索引组织表（IOT）
	页分裂：
		页可以为空，也可以填充一半，也可以完全填充，每个页如果插入数据，那么至少要插入两行数据（如果插入数据过大会产生行溢出），插入的数据根据主键排列（其实是根据聚集索引排列，不过聚集索引一般是主键）
		特别说明一点：
			下方的插入是指将数据插入到创建好的B+Tree时，此时树的索引顺序已经确定，下方讲的插入是将数据插入到索引后的页中。
		主键顺序插入时的情况：
			首先数据库会申请一个16k的储存空间作为页来存放数据，数据按照主键顺序向页内进行插入，当一个页满后，就会申请下一个页来存放数据，此时，这两个页之间会存在一个双向指针，第二个页也写满后以此类推。
		主键乱序插入时的情况（可能产生页页分裂）：
			这里的乱序指的是插入数据的顺序混乱，但插入后数据依旧是顺序排列的。
			假设目前已经插入了两个页，第一个页的索引为：1,5,9,12,16。第二个页的索引为：23,26,47,95。两个页均已存满数据，此时再向数据库中插入一个索引为20的数据，这个数据应该加在第一个页的索引16后面，但此时储存已满，因此数据库会将第一个页从50%的地方分裂，然后将20加在分裂后的第三个页中，因为此时页中索引的顺序不连续，因此数据库会调整页的位置，将第三个页放到第二位，并重新设置链表指针，这就是页分裂现象，因此，乱序插入的性能要低于顺序插入。
	页合并：
		当删除数据时，实际上数据并没有被物理删除，不过被标记了，被标记的空间可以被其他数据再次声明使用。
		当页中被删除的记录达到一个阈值（默认50%）时，InnoDB引擎会自动寻找最近的页，然后查看是否能将两个页进行合并（两个页加起来使用空间不超过100%），如果可以就会合并两个页以优化空间使用，这就是页合并。
		merge_threshold:合并页的阈值，默认50%，在创建表或索引时可进行指定
	主键设计原则：
		1.在满足业务需求的前提下尽量降低主键长度
			因为其他二级节点的叶子节点中挂的是数据的主键，如果主键较长，二级索引较多就会占用大量磁盘空间，搜索时浪费大量磁盘IO
		2.插入数据时尽量选择顺序插入，选择auto_increment这种自增主键
		3.尽量不要使用uuid或其他自然主键，如身份证号
			这种主键插入时会乱序插入，而且长度较长
		4.业务操作时尽量避免对主键的修改
order by优化（排序优化）：
	有两种排序方法：
		using filesort：
			通过表的索引或全表扫描，读取满足条件的数据行，然后在排序缓冲区sort buffer中完成排序操作，而不是通过索引直接返回排序结果的排序都叫filesort排序
		using index：
			通过有序索引扫描直接返回有序数据，这种情况即为using index，不需要额外排序，操作效率高
	优化原则：
		1.根据排序字段建立合适的索引，多字段排序时，也遵循最左前缀法则
		2.尽量使用覆盖索引
			不使用覆盖索引的话查询出数据后还要进行回表查询，查询出的数据再进入排序缓冲区进行排序
		3.多字段排序，一个升序一个降序，此时需要注意联合索引在创建时的规则
			索引在创建时也有升序索引和降序索引的区别，如果多字段查询一个升序一个降序，而索引都是升序那么就需要在排序缓冲区进行排序
		4.如果不能避免出现filesort，大数据量排序时可适当增加缓冲区大小，使用sort_buffer_size控制，默认大小256k
			如果排序数据大小超过排序缓冲区大小，那么就会在磁盘中进行排序，效率低于在排序缓冲区排序
group by优化（分组优化）：
	可以通过建立索引优化分组查询的速度
	分组查询时，索引使用也满足最左前缀法则
limit优化（分页优化）：
	当分页查询仅查询前面的几条数据时，查询速度很快，但查询的数据位置越靠后，查询速度越慢，因为分页查询会查询并排序从开始到指定输出的所有数据，但不要求显示的数据会被丢弃，例`select * from table_name limit 100000 10;*`会查询100010条数据，然后丢弃前10000条。
	通过覆盖索引+子查询，不过MySQL不支持直接将limit查询的语句作为子查询，例如：
	`select * from table_name where id in (select id from table_name limit 10000 10);`这种查询方式，要使用多表查询的方法，将limit查询到的数据作为新表来进行处理，例如：`select s.* from table_name s ,(select id from table_name limit 10000 10) a where s.id = a.id;`这种方法进行查询
count优化：
	MyISAM引擎把一个表的总行数存在磁盘上，因此在执行count（`*`）的时候会直接返回这个数，效率较高
	InnoDB引擎在执行count（`*`）的时候会一行一行读取数据然后计数，效率很低
	count的几种方法，按效率从高到低
		count(`*`)
			InnoDB引擎专门做了优化，不会进行取值，而是直接在服务器层按行进行累加，因此效率最高
		count（数字）
			对每一行进行一个常量计算，勉强可以理解为1=1吧，然后就会返回这个常量，因为返回值必定不为null,因此每一行都会被计数，因为也不对表进行取值，只进行简单计算，因此效率与count（`*`）相近。
		count(主键)
			会将每一行的主键值取出，然后返回给服务层，服务层拿到主键后按行进行累加（主键不会为null）
			因为多了个取值，所以效率较低
		count（字段）
			没有not null约束：
				会将每一行的字段值取出给服务层，服务层判断是否为null后进行计数，效率最低
			有not null约束：
				与没有not null约束相比，不用判断是否为null，效率略高，如果存在索引，且索引结构，大小，扫描路径与主键索引相同时，二者效率相同，如果没有索引，那么count（主键）效率更高，因为主键必定有索引
undate优化：
	更新字段时要根据索引进行更新，因为InnoDB引擎使用行级锁，在执行事务时会将行锁住阻止其他用户变动改行，如果没有索引，那么行锁会升级为表锁，将整个表都锁住。
